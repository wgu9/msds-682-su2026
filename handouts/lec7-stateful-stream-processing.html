01
Define stream processing

Computation begins while the event history is still growing

  • Continuous input: events keep arriving.
  • Incremental work: process new records as they appear.
  • New facts: emit transformed, aggregated, or joined results.

Responsibility boundary

Kafka stores the history; a processor interprets it

Kafka broker

Retain and deliver

Topics, partitions, offsets, replication, replay.

Processor

Compute and remember

Filter, transform, aggregate, join, classify, and emit.

Classroom scripts stay bounded for evidence; the production processor keeps polling as new records arrive.

Kafka is not the business computation. Client code or a processing framework performs that work.

The first design question

Does one input record contain everything you need?

PatternNeeds memory?Example
StatelessNo prior record requiredValidate one order and compute one metric
StatefulMust remember earlier recordsWait for a fare quote and its later trip outcome
State begins when one record is not enough.

Operation map

Four operations cover most teaching examples

Filter

Keep?

One record in, zero or one out.

Map

Transform

One record becomes a new record.

Aggregate

Accumulate

Many records update one result.

Join

Relate

Records from two histories become one fact.

Filter and map are often stateless. Aggregate and join normally require state.
02
Model the data

Keys and time determine what can be remembered

  • Stream: the history of events.
  • Table: the latest value by key.
  • Time: when an event happened and how long state remains useful.

Two views of changing data

A stream records changes; a table represents current state

Stream

Every event matters

trip-requested, fare-quoted, trip-completed. History is append-oriented.

Table

Latest value by key

trip_id → latest trip status. Later updates replace the current view.

Stream and table are interpretations. Kafka topics remain ordered event logs.

Keyed computation

The key decides which events belong together

trip_id=682-AFare quote
→
Aligned partitionsSame key and compatible partitioning
←
trip_id=682-ADelayed outcome
A shared key is necessary for a keyed join, but not sufficient. Contracts, time bounds, state, and recovery policy still matter.

Demo 07 uses one partition per topic. Larger systems must align partition count and partitioning strategy across joined inputs.

Two clocks

When did the event happen, and when did we process it?

ClockMeaningWhy it matters
Event timeTime recorded by the business eventWindows should reflect when the activity occurred
Processing timeTime the processor sees the recordNetwork delay and replay can move it later
Demo 07 uses a finite classroom join. Production systems must state which clock governs expiry.

Bounded state

A join window bounds how long related records may arrive

OpenFirst side arrives; store keyed state
→
MatchOther side arrives within policy
→
ExpireNo match by the declared boundary
A late event is a policy decision. Drop, update, compensate, quarantine, or reprocess. The business must choose.
03
Stateful joins

Subscribing receives records; joining creates a relationship

  • Store the unmatched side by key.
  • Match when the other side arrives.
  • Expire or recover state under an explicit policy.

Common misconception

Two subscriptions do not automatically produce one joined fact

Subscribe

Receive both histories

The consumer sees quote records and outcome records.

Join

Implement correlation

Extract key, store unmatched input, match, compute, emit, and clean up.

The application owns the join semantics.

Join lifecycle

One key moves through four explicit states

1 · ReceiveDecode and validate
→
2 · RememberStore by trip_id
→
3 · MatchQuote + outcome
→
4 · EmitPricing evaluation
State is not the output. It is temporary information required to produce the output.

A join needs more than a key

Write the policy before writing the loop

QuestionDemo 07 answer
What matches?Fare quote and trip outcome with the same trip_id
How many results?One evaluation per quote version and trip
When is input complete?Eight expected quote-outcome pairs in the bounded run
What is late or missing?Reported as a production gap, not silently invented

Crash boundary

In-memory state teaches the join, not production recovery

Classroom

Finite dictionaries

Transparent and easy to inspect.

Failure

Memory disappears

A restart must rebuild or restore state.

Production

Durable state

Checkpoint, changelog, replay, or managed state store.

Bounded stateful teaching code is not a durable stream processor.

Build one pricing pipeline from request to delayed evaluation

  • Estimate distance and duration.
  • Quote with a rule and a trained model.
  • Evaluate both versions after the outcome arrives.

Ridesharing pricing case

Set an upfront fare before actual trip cost is known

Trip requestRequest time + pickup/dropoff coordinates
→
Routing responseEstimated miles + minutes
→
Fare quoteFixed before the trip begins
→
Delayed outcomeSynthetic realized miles, minutes, and cost
$$\text{realized markup} = \frac{\text{fare} - \text{actual cost}}{\text{actual cost}}$$

Business target: realized markup close to 20%.

Inputs, estimates, and outcomes

Request

Predefined coordinates

Four fixed latitude/longitude pairs in decimal degrees.

Quote time

Fixture or OSRM

Meters/seconds become miles/minutes. Fixture is controlled; OSRM is live.

After pricing

Synthetic outcome

07D varies miles/minutes and computes synthetic cost in cents.

07D has no GPS and does not wait for a real trip. It is an independent bounded outcome source.

Two pricing versions

Compare a transparent rule with a trained cost model

VersionComputationRole
rule-v1USD fare = 3.50 × estimated miles + 0.20 × estimated minutesHand-calculable heuristic baseline
ridge-v2USD fare = predicted cost × 1.20Trained model plus explicit markup policy
The rule is not ML. Ridge is a small, stable teaching model, not a claim about the best production pricing model.

Architecture decision

Keep fields together when one owner computes them together

DesignPipelineConsequence
A · BuiltOne route call → miles + minutes → quoteNo artificial join

The meaningful join is quote plus delayed outcome, not split route fields.

05
Evaluate and bound

A model comparison needs shared outcomes and one declared metric

  • Measure both versions on the same trips.
  • Select the lower-error version under the stated rule.
  • Separate recommendation from production deployment.

One business metric

Compare absolute distance from the 20% markup target

$$\text{MAE}_{\text{markup}} = \operatorname{average}\bigl|\,\text{realized markup} - 20\%\,\bigr|$$

Lower error is better. The report expresses the result in percentage points.

Fair comparison

Same evidence

Both versions use the same requests, routes, and delayed outcomes.

Selection rule

Lower target error

Recommend the version with the smaller reported error.

Governance boundary

A recommendation is evidence, not a deployment

EvaluateCompute the declared metric
→
RecommendPrefer the lower-error version
→
ReviewCheck sample size, risk, and limits
→
Deploy laterOutside Demo 07
Model promotion is the production act of approving a version. Demo 07 stops at a selection recommendation.

Lecture 7 synthesis

Five ideas organize stateful streaming

  1. Kafka retains history; the processor computes meaning.
  2. State begins when one record is not enough.
  3. Keys and time define what may relate.
  4. Joining stores, matches, emits, and expires.
  5. A prediction needs a later outcome to evaluate.
