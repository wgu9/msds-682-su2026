Unit

Table stores rows. Topic stores messages (events).

Change

Rows are updated in place. Messages are immutable — change = new message.

Order

Topic is temporal: appended at the end, ordered within a partition.

Read pattern

A query asks for current state. A consumer replays events from an offset.

A topic is an append-only log

A ridesharing trip_events topic: one trip's lifecycle, appended in time order — never rewritten.

Key point: each message = value (business data) + metadata (key, timestamp, headers from the producer; offset assigned by Kafka). All three events share key: trip_981, so one trip's events land in the same partition, in order.

No enforced schema

Kafka stores bytes. Messages in one topic do not need identical fields — trip_requested and driver_matched carry different payloads.

One logical stream

What shares a topic is a business decision: everything in trip_events is about trips. Payments and ratings get their own topics.

Schema Registry note

In our homework (Avro + Schema Registry), each topic's value maps to one subject whose schema evolves by version — a team convention, not a Kafka rule.

Broker

A server in the cluster. It stores partitions — not whole topics — and serves client requests.

Leader

Each partition has one leader. All reads & writes for that partition go through it.

Replica

Copies on other brokers. If Broker 1 dies, the replica of P0 on Broker 2 is promoted to leader.

Segment

On disk, a partition = append-only files. Retention deletes old segments, not individual messages.

Kafka order = partition offset order, not timestamp order

A real question from class: "Every event already carries a timestamp — I can just sort later. So what does Kafka's time order even mean?"

The mailbox analogy
Postmark · event timestamp

Self-reported time, stamped by the sender when the event is created. Clocks drift, phones misreport, delivery gets delayed — the postmark can lie.

Mailbox order · offset

The order letters actually land in the box. Assigned by Kafka at write time — monotonic, immutable, never re-sorted.

Five arrivals, one partitiontrip_events–P0 · cells show event timestamps
offset 010:00
offset 110:01
offset 210:04
offset 310:02
offset 410:05

The event stamped 10:02 arrived late — Kafka still appends it at offset 3. Consumers always read 0 → 1 → 2 → 3 → 4; the log is never reordered.

Key point: Kafka's promise is the written order within a partition — strict, immutable, defined by offset. The timestamp inside the event is just a field: useful for analytics, not proof of who came first.

Postmark vs. offset — the full mapping

"Think of the timestamp as the postmark, and the offset as the mailbox order. A letter can carry an earlier postmark but arrive later."

Mail ↔ Kafka

the analogy, term by term

Mail systemKafka
Postmark timeevent timestamp (event_time)
Arrival order at the mailboxwrite order into a partition
Position in the stackoffset
Postal delays, trafficnetwork latency, retries, buffering
Early postmark, arrives lateearly event_time, larger offset
Late postmark, arrives earlylate event_time, smaller offset

Letters A & B

same partition

A · postmark 10:00arrives 10:05
B · postmark 09:58arrives 10:08
offset 0A · 10:00
offset 1B · 09:58

B's postmark is earlier — yet B sits at offset 1. Arrival decided the position, not the stamp. Kafka records exactly this.

Conclusion: the postmark doesn't decide the mailbox order, and the timestamp doesn't decide the offset. Kafka's only promise: within a partition, first written ⇒ smaller offset.

② Check first, then create

demo01_create_topic.py — block 2 of 3

def topic_exists(admin_client: AdminClient, topic_name: str) -> bool:
    return topic_name in admin_client.list_topics(timeout=10).topics

def create_topic(
    admin_client: AdminClient,
    topic_name: str,
    partitions: int,
    replication_factor: int,
    cleanup_policy: str,
) -> str:
    if topic_exists(admin_client, topic_name):
        return "already_exists"

    topic = NewTopic(
        topic_name,
        num_partitions=partitions,
        replication_factor=replication_factor,
        config={"cleanup.policy": cleanup_policy},
    )
    futures = admin_client.create_topics([topic])
    futures[topic_name].result(timeout=30)
    return "created"
Idempotent Check before create — re-running returns already_exists instead of crashing.
Three knobs of NewTopic partitions = parallelism · replication_factor = fault tolerance (class default: 3; allowed values depend on cluster settings) · cleanup.policy = delete (expire by retention) or compact (latest per key).
Async under the hood create_topics() returns futures; result() blocks until the broker confirms — or raises.

Sync vs async producer

Demo 01 left msds682.demo01.trip-events.v1 empty — 02A–02D now produce real trip events into it. The real Kafka default is async; sync-style is a teaching simplification.

# demo02c_confluent_async_sync_compare.py
def run_async(config, count, seed, flush_timeout):
    producer = Producer(config)
    tracker = DeliveryTracker()
    events = make_trip_events(count, seed)
    for event in events:
        # Async default: enqueue all messages first.
        producer.produce(TOPIC_NAME, key=event_key(event),
                         value=serialize_event(event),
                         callback=tracker.callback)
        producer.poll(0)
    remaining = producer.flush(flush_timeout)

def run_sync_style(config, count, seed, flush_timeout):
    producer = Producer(config)
    tracker = DeliveryTracker()
    events = make_trip_events(count, seed)
    remaining = 0
    for event in events:
        # Teaching simplification: wait after each produce.
        producer.produce(TOPIC_NAME, key=event_key(event),
                         value=serialize_event(event),
                         callback=tracker.callback)
        remaining = producer.flush(flush_timeout)
Real default: async produce(...) queues the message and returns quickly. Delivery result arrives later through tracker.callback.
poll(0) serves callbacks It lets the client run delivery callbacks while the loop continues. It is not Python asyncio.
flush() is the wait point Async path: queue all messages, then flush once before exit. Sync-style path: flush after each message, only for comparison.
Do not teach sync as Kafka default Demo 02A and Demo 02C use sync_style_flush_each_message. It is slower and simplified, not the normal producer pattern.

02A sync-style: wait after every message

produce 2

Four messages, four blocking waits. Slowest, but easiest to reason about.

02B async: queue first, wait once at the end

Four messages, one final wait. Network delivery can overlap while the loop continues.

# demo02_producer_common.py
class TripEvent(BaseModel):
    trip_id: str
    event_type: Literal["trip_requested", "driver_matched",
                        "trip_started", "trip_completed"]
    rider_id: str
    event_time: str
    zone: str
    driver_id: str | None = None
    fare: float | None = Field(default=None, ge=0)

def event_key(event: TripEvent) -> bytes:
    return event.trip_id.encode("utf-8")

def serialize_event(event: TripEvent) -> bytes:
    return event.model_dump_json(exclude_none=True).encode("utf-8")

# demo02d_confluent_serialization_producer.py
value_bytes = serialize_event(event)
producer.produce(
    topic=TOPIC_NAME,

Why async wins: the client batches internally

True or false — the producer sends messages one by one to the broker? False. produce() appends to a local buffer; whole batches cross the network.

Your code            Producer library (in-process)      Broker
   |                          |                            |
produce(msg 1) ---> [ buffer: 1       ]  returns           |
produce(msg 2) ---> [ buffer: 1 2     ]  immediately;      |
produce(msg 3) ---> [ buffer: 1 2 3   ]  local append,     |
produce(msg 4) ---> [ buffer: 1 2 3 4 ]  no network        |
   |                          |                            |
   |    batch closes when ANY limit is hit FIRST:          |
   |      time : linger.ms          = 5 ms                 |
   |      size : batch.size         = 1,000,000 bytes      |
   |      count: batch.num.messages = 10,000               |
   |                          |                            |
   |     [ Batch: msg 1-4 ] == ProduceRequest ==>  append  |
   |                          |                   to log   |
   |                          | <====== ack ======         |
   |                          |                            |
poll(0) -> serve ready callbacks; never waits              |
flush() -> serve callbacks until the queue drains          |
Not one by one Only ProduceRequests carrying record batches cross the network. A quiet partition may ship a one-record batch — it is still a batch.
librdkafka 2.15 defaults — our demos do not tune them linger.ms=5, batch.size=1,000,000 bytes, batch.num.messages=10,000. Whichever limit hits first closes the batch. (The Java client differs: 16 KB default batch.size, no count trigger.)
poll(0) vs flush(), precisely poll(0) runs callbacks that are already ready and never waits for acknowledgements. flush() keeps serving callbacks until every queued and in-flight message completes or the timeout expires.
Three “batch sizes” — do not mix them ① Client config batch.size above: defaults, untouched in this course. ② Assignment 1’s --batch-size 500: an application-level measurement group — async flushes at each group boundary, sync-style flushes every message. It is not Kafka batch.size. ③ Demo 02 exposes no batch flag at all — batching happens silently inside the client.

Direct rule: batching is why the real run three slides back shows 1,893.55 vs 22.53 messages/sec. Async lets the client fill batches while your loop continues; flushing after every message forces one full network round trip per record.
