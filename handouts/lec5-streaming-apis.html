Why use Schema Registry if we already have Pydantic?

Inside one application

Pydantic validates meaning

It checks Python objects, field constraints, and business rules at an application boundary.

Across applications

Schema Registry governs the wire contract

It coordinates schema IDs, versions, and compatibility across independently deployed producers and consumers.

A local schema file is only a copy. By itself, it cannot coordinate deployed versions or compatibility.
Use both: Pydantic owns application rules; Avro plus Schema Registry own the shared Kafka wire contract. The registry stores schemas, not records.

Where does validation happen from producer to consumer?

Producer appPydantic checks input.
→
Avro serializerEncodes declared fields and a schema ID.
→
KafkaRetains bytes.
→
Consumer appDecode, validate, process, then commit the consumer offset.

What can fail?

  • Missing fields or wrong types can fail before publish.
  • Unknown schema IDs or corrupt bytes can fail during deserialization.
  • The course flow commits only successfully processed records.

What if the object has 15 fields but Avro defines 5?

  • Avro is not an automatic filtering policy.
  • Our pinned writer may omit undeclared dictionary keys.
  • Reject them before publish with strict Pydantic extra="forbid".
Compatibility is checked when a new schema version is registered, not for every message. Kafka normally retains bytes without field-level validation.

The new boundary

A web API receives commands; Kafka records durable facts

The two interfaces serve different jobs and should remain visible.

HTTP command

“Request a trip”

  • short request and response lifecycle;
  • client-facing validation and status codes;
  • may contain intent that is not yet a durable fact.

Kafka event

“Trip was requested”

  • durable append-only record;
  • independent consumers can process it later;
  • includes event identity, time, key, and governed value.
KEY TAKEAWAY: FastAPI accepts an application command; Kafka publishes the durable event created from it.

HTTP semantics

Status codes describe what the application can honestly claim

StatusUse in this lectureWhat it means

Do not return success too early. “Python function returned” and “Kafka acknowledged the event” are different states.

Application lifespan

One service should own one long-lived producer

from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(app: FastAPI):
    publisher = await create_publisher()
    app.state.publisher = publisher
    try:
        yield
    finally:
        await publisher.close()

app = FastAPI(lifespan=lifespan)

Delivery semantics

“Accepted” is not one moment

1. Request validJSON matches the HTTP model.
2. Event validThe core built TripEventV1.
3. EnqueuedThe local producer queue accepted it.
4. AcknowledgedKafka returned topic, partition, and offset.
5. ProcessedThe consumer completed business work.
6. CommittedThe group stored its recovery position.
Course decision: wait through broker acknowledgement before returning 202 Accepted. The response still does not promise downstream processing.
