Why free-form JSON becomes fragile

An implicit contract works until teams deploy independently

JSON syntax can be valid while the producer and consumer disagree on field name, type, or meaning.

# producer version A
{
  "trip_id": "trip_1001",
  "fare": 23.40,
  "event_time": "2026-07-16T17:00:00Z"
}
# consumer silently assumes
{
  "tripId": "...",
  "fare": "23.40",
  "event_time": epoch_seconds
}
Valid JSON is not enough. Without an explicit contract, drift may appear only after production data reaches a consumer.

Contract thinking

A schema is a contract between producers and consumers

The contract is useful because it makes assumptions inspectable, testable, versioned, and rejectable before deployment.

Namestrip_id, not an undocumented alias
Typesstring, enum, timestamp, nullable double
Presencerequired fields and fields with defaults
Evolutionwhich old/new reader-writer pairs resolve
Governancewho may register or reject a version
Schema does not update application code. It gives the team a compatibility gate and the client enough structure to decode bytes.

Why Avro in this lecture

Avro makes schema evolution visible

The learning objective is the writer/reader model, not claiming that every production team must choose Avro.

Compact

Field names are not repeated in every binary record.

Typed

The reader knows whether data is a string, enum, logical timestamp, union, or double.

Resolvable

Writer and reader schemas can differ under defined rules.

Registry-friendly

A small schema ID connects Kafka bytes to a versioned schema service.

Avro anatomy

Read the schema as a record contract

Focus on names, types, logical types, unions, and defaults.

{
  "type": "record",
  "name": "TripEventV1",
  "namespace": "edu.usf.msds682",
  "fields": [
    {"name": "trip_id", "type": "string"},
    {"name": "event_type", "type": {"type": "enum", ...}},
    {"name": "event_time",
     "type": {"type": "long", "logicalType": "timestamp-millis"}},
    {"name": "fare", "type": ["null", "double"], "default": null}
  ]
}

Full name

edu.usf.msds682.TripEventV1

Enum

Finite event symbols prevent arbitrary strings on the wire.

Logical type

A numeric encoding with timestamp meaning.

Union + default

Supports nullable data and compatible reader resolution.

Optional fields

Nullable, omittable, and defaulted are related, not identical

Be precise about what the writer sends and what a reader does when a field is absent.

Nullable union

"type": ["null", "string"]

The value may be null or a string. Union branch order matters when a default is supplied.

Reader default

"default": null

A reader can supply null when an older writer schema did not contain the new field.

Do not teach: “adding default makes every producer rule optional.” The active writer schema and application model still determine what may be written.

Confluent wire format

The payload references a schema ID

Default framing keeps each record small while allowing the consumer to retrieve the writer schema.

magic byte
0
schema ID
32-bit integer
Avro binary body
record encoded with the writer schema

Not included

The full schema text is not repeated in every Kafka record.

Consumer action

Read schema ID, use cache or Registry, then decode the Avro body.

Kafka action

Store and return the bytes without interpreting the header.

Schema Registry

Registry manages schemas, not event data

Kafka holds records. Registry holds contracts and governance metadata.

Schema Registry stores

  • schema text and schema type;
  • subject and version history;
  • schema IDs and lookups;
  • compatibility configuration;
  • references, metadata, and rules where used.

Schema Registry does not store

  • Kafka topic records;
  • consumer offsets;
  • producer delivery acknowledgements;
  • business processing results;
  • automatic application-code updates.

Registration path

Registration is not a per-record full-schema upload

The serializer registers or looks up the schema, remembers the result, and writes the ID into subsequent values.

1Compute subject from topic and field
2Register or look up writer schema
3Receive schema ID
4Cache subject/schema metadata
5Frame and encode each record

Compatibility

Compatibility is a deployment gate

Before accepting a new schema version, Registry compares it with the configured history.

New schema proposedsubject version N+1
→
Compatible → registerClients can deploy under the chosen reader/writer promise.
Incompatible → rejectChange the schema, migration, subject, or rollout plan.
Compatibility is structural. A type-preserving change in business meaning may pass the Registry and still break users.

Compatibility modes

Ask who must read whose data

Backward and forward name reader/writer relationships.

ModeQuestionTypical rollout implication
BACKWARDNew reader reads previous writer data?Readers first, then writers.
FORWARDPrevious reader reads new writer data?New writers may coexist with old readers.
FULLDo both directions resolve?Stricter mixed-version coexistence.
TRANSITIVECompare against all earlier versions?Protect longer retention and replay.

Inspect the effective subject setting; do not assume a default.

Writer and reader schemas

The bytes were written with one schema and may be read with another

Avro resolution is what makes compatible evolution concrete.

Writer schema V1

trip_idstringrequired
event_typeenumrequired
farenull | doubledefault null

Reader schema V2

trip_idstringfrom writer
event_typeenumfrom writer
farenull | doublefrom writer
vehicle_typenull | stringreader default null
Result: a V2 reader can reconstruct vehicle_type=None from a V1 record that never contained that field.

Safe evolution example

Add an optional field with a usable default

This is the version-1 writer / version-2 reader case used by Demo 04B.

// V2 reader addition
{
  "name": "vehicle_type",
  "type": ["null", "string"],
  "default": null
}

Old data

No vehicle_type bytes were written.

New reader

Uses the declared default while resolving the record.

Application

Receives a typed V2 object with vehicle_type=None.

Breaking changes

Not every field edit is an evolution

Unsafe changes need migration, rollout coordination, or a new contract.

ChangeRiskWhy
Required field, no defaultbreakNew reader cannot rebuild old data.
Rename without aliasbreakReader sees a missing field.
Remove a retained enum symbolbreakNew reader cannot resolve it.
string to unrelated recordbreakNo resolution path.
Optional field + matching defaultusually safeReader supplies the older missing value.

Semantic change

A compatible type can still carry incompatible meaning

Registry cannot infer every business invariant.

Structurally valid Avro

"fare": -10.0

The Avro field type is double. Negative is still a double.

Application-invalid event

fare = Field(ge=0)

Pydantic rejects the same decoded record before business processing or commit.

Contract stack: Registry compatibility + Avro structural decoding + application validation + business processing.

They solve different problems and reinforce each other.

Python models in 2026

Dataclass remains valid; Pydantic is the course boundary model

The choice is about validation needs, not whether one feature is “modern.”

dataclass

Lightweight domain data

  • @dataclass(kw_only=True) removes positional default-order pressure.
  • slots=True can reduce instance overhead.
  • Validation and JSON parsing require additional code or libraries.

Pydantic v2

Boundary validation

  • Strict JSON parsing and rich validation errors.
  • Timezone-aware datetime types and model validators.
  • Useful before serialization and after deserialization.
Demo 04 decision: Pydantic owns application validation; Avro owns the cross-language wire schema.

Production/consumption separation

Transmit source facts; derive reusable views downstream

A consumer does not need every analytical field serialized into the event.

Producer sends

  • trip_id
  • event_type
  • event_time as an aware instant
  • zone, optional driver and fare

Consumer derives

  • event_date
  • event_hour
  • event_weekday
  • window, feature, or warehouse partition fields
Data enrichment: derived fields stay consistent with the source timestamp and do not bloat every event.
