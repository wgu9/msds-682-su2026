01
Choose the boundary

Start with the responsibility, not the tool

  • Integration moves records between systems.
  • Application I/O implements product behavior.
  • Computation derives facts and state.

Decision rule

Choose the smallest tool that owns the job

You need toStart with
Move records from a supported external systemKafka Connect
Implement an HTTP or domain applicationNative client
Filter, aggregate, classify, or join with stateStream processor
Configuration-first is not code-free. Custom connectors and transforms still require engineering.

One common confusion

Three interfaces serve three audiences

Business API

FastAPI

Clients send commands, get responses.

Control plane

Connect REST API

Operators create, pause, and inspect connectors.

Data plane

Kafka records

Systems exchange durable events.

FastAPI is the HTTP boundary; its Kafka Producer client writes records. Connect REST only manages connector lifecycle.

04
Stream processing

Turn retained events into new facts

  • Kafka retains; the processor computes.
  • Demo 06C is stateless; stateful joins come next.
  • Acknowledge output before committing input.

Two processing shapes

State begins when one record is not enough

Stateless

One record in, one decision out

Validate, redact, convert units, enrich from fixed configuration.

Stateful

Output depends on prior records

Count, deduplicate, window, join, maintain a table.

Demo 06C is stateless. Lecture 7 adds keyed state, bounded windows, and joins.

Delivery boundary

Commit input only after output is acknowledged

ConsumeRead one input record
→
ValidateWire, then meaning
→
ProcessDerive the result
→
Output ackBroker confirms delivery
→
Consumer commitStore the next input offset
Commit means consumer offset commit. Not producer delivery. Not Git.

Failure reasoning

This ordering gives at-least-once, not exactly-once

The crash window

Output delivered, commit missed

After restart the same input is read again and may produce again.

The control

Stable event identity

A deterministic key exposes duplicate identity and enables downstream deduplication.

Do not promise exactly-once from commit order alone. Transactions have extra requirements.
Reference: Confluent Python transactional API

Demo 06D · Recovery behavior

Resume continues; replay revisits history

Resume · same group

Start after the committed offset

Normal recovery: the group picks up where it stopped.

Replay · new group

OFFSET_BEGINNING in on_assign; the base group is untouched.

This transformation has no application state. Stateful replay must also rebuild derived state.

Lecture 6 synthesis

Five ideas organize the whole system

  1. Connect moves records between systems.
  2. Kafka retains records ordered within each partition.
  3. A processor computes and may own state.
  4. Output acknowledgement precedes input commit.
  5. Resume continues; replay revisits history.
