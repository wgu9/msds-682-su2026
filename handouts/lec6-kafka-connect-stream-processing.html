Choose the boundary

Start with the responsibility, not the tool

IntegrationMove records between systems.
Application I/OImplement product-specific behavior.
ComputationDerive facts, state, or outputs.

The integration problem

Copying data should not require a new application every time

Common systems already expose recognizable source and sink patterns.

Database

Rows change

Capture records and publish them to Kafka.

Object storage

Files arrive

Move records into or out of Kafka on a repeatable schedule.

Analytics system

Events need a destination

Deliver topic records without embedding sink code in producers.

Kafka Connect standardizes the plumbing. It does not own arbitrary product decisions.

Decision table

Choose the smallest tool that owns the job

NeedBest starting pointWhy
Move records from a supported systemKafka ConnectConfiguration-first integration with managed lifecycle and offsets.
Implement an HTTP or domain applicationNative clientThe application owns validation, responses, and product behavior.
Filter, aggregate, classify, join, or maintain stateStream processorThe computation has explicit keys, state, outputs, and recovery rules.
Use the smallest transformation boundary. SMTs fit lightweight record-at-a-time changes. State, windows, joins, and business decisions belong in a processor.

Operational evidence

Status and offsets answer different questions

Connector statusIs the connector instance running, paused, or failed?
Task statusWhich execution units are healthy or failing?
Source offsetWhere has a source connector reached in the external system?
Consumer offsetWhere has a sink or processor consumer group reached in Kafka?
Do not collapse these into one “offset.” Each offset belongs to a specific progress boundary.

Failure handling

A running connector can still contain record-level failures

Inspect separately

  • connector state: is coordination healthy?
  • task state: did one execution unit fail?
  • record errors: did conversion or destination writing fail?
  • lag or stalled offsets: is progress continuing?

Policy

Fail, retry, log, or tolerate

Capabilities vary by connector and direction.

DLQ is not universal. Do not promise that every source or sink connector handles bad records the same way.

Turn retained events into new facts

ComputeFilter, transform, aggregate, or classify.

Responsibility boundary

Kafka retains records; the processor gives them new meaning

Kafka broker

Durable transport

  • ordered partition logs;
  • keys, values, headers, offsets;
  • retention and independent consumer groups.

Stream processor

Application computation

  • validation and transformation;
  • state, windows, and business rules;
  • derived records written to another topic.
Kafka is the backbone, not the business computation.

Two processing shapes

State begins when one record is not enough

Stateless

One record in, one decision out

Examples: validate, redact, enrich from fixed configuration, convert units.

Stateful

Current output depends on prior records

Examples: count, rolling feature, deduplication, table update, windowed classification.

State needs an owner. Define its key, lifetime, recovery path, and output contract.

Two ways to interpret records

A stream preserves events; a table represents current keyed state

ViewQuestionExample
StreamWhat happened, and in what order?Trip requested, driver assigned, trip completed
TableWhat is the latest known value for this key?Current trip status by trip_id
The same Kafka records can support both interpretations. The processor defines how state is materialized.

Stateful processing ingredients

Key, state, and window make correlation explicit

KeyWhich entity owns this sequence?
+
StateWhat prior information must survive?
+
WindowWhich bounded interval belongs together?
Subscribe is not a join. Correlation requires explicit keys, state, time boundaries, and missing-data rules.

Delivery boundary

Commit input only after the required output is acknowledged

ConsumeRead one input record
→
ValidateWire + application checks
→
ProcessCompute the derived result
→
Output ackBroker confirms delivery
→
Input commitStore next consumer offset
Commit means consumer offset commit. It is not producer delivery acknowledgement and not Git commit.

Failure reasoning

Correct ordering reduces loss, but duplicates are still possible

Crash point

Output succeeded; input commit did not

After restart, the consumer reads the same input again and may produce the output again.

Baseline

At-least-once

Prefer possible duplicates over silently losing completed work.

Control

Idempotency

Use stable event identity and make repeated processing safe.

Do not promise exactly-once from commit order alone. Transactions and connector or framework support have additional requirements.
Reference: Kafka Connect exactly-once support and limitations

Demo 06D · Recovery behavior

Resume and replay answer different questions

Resume

Use the same group

Kafka starts after the group's committed offset. This proves normal recovery.

Replay

Use an explicit replay group

A distinct group sets assigned partitions to OFFSET_BEGINNING without changing the base group's progress.

This transformation has no application state. A future stateful replay must reset or rebuild derived state as well as input offsets.
