Q1. Do we need “Skills” in Codex or Claude Code?

Thanks to everyone who asked questions related to our course.

Short answer: not on day one; yes once you repeat yourself.

Start with project context

A skill is packaged guidance that an agent loads on demand. The highest-leverage first step is a useful CLAUDE.md or AGENTS.md explaining how the repository is built, tested, and structured.

Rule of thumb: the third time you paste the same instructions into a session, turn them into a skill.

Choose what deserves reuse

  • Good candidates: stable, repeatable workflows.
  • Bad candidates: one-off tasks; an unused skill becomes maintenance debt.
  • For SOTA models: post-training has commoditized much of the “how.” Procedure-only skills can regress as models improve; skills that declare intent and facts appreciate.
  • Evaluate the skill: evidence is how you distinguish the two.
What remains yours: private context, contracts, constraints, and the judgment to verify the result.

Q2. How should we manage subagents?

Rule of thumb: use isolated worktrees when multiple agents write to the same repository. Avoid conflicts.

Treat a subagent like an intern with no memory of your conversation: it starts cold, so the brief determines the result.

1–2 · Delegate deliberately

  • Define a deliverable. “Find every hardcoded topic name and report file:line” works; “help with the refactor” does not.
  • Use fan-out selectively. Parallel searches, noisy test runs, and fresh review passes benefit; homework-scale work usually benefits more from one agent with strong context.

3–4 · Control and verify

  • Keep write access visible. Let the main agent own writes; make research subagents read-only when possible.
  • Verify before acting. A subagent summary is a claim, not a fact—the same evidence discipline required by AI_USAGE.md.

Restored · producer batching 1 of 3

True or false: each message crosses the network alone?

False. The Python call handles one record; the client transports record batches.

Your Python loop

produce(message)

  • The application prepares one serialized key/value record.
  • Appends it to a local, partition-specific queue.
  • Usually returns before the broker acknowledges delivery.

The network request

One ProduceRequest carries batches

  • A background librdkafka thread builds a batch per partition.
  • A quiet partition may send a one-record batch—it is still a batch.
  • Compression works on batches, not on isolated Python function calls.
Why the sync-style benchmark is slow: calling flush() after every produce() prevents useful accumulation and forces the application to wait for each record’s network completion.

Restored · producer batching 2 of 3

Time, bytes, and count bound batch accumulation

librdkafka waits up to 5 ms to accumulate records unless the size or count limit is reached first.

Your code          Producer library                  Broker
   |                       |                           |
produce(1) ------> [ partition buffer: 1       ]      |
produce(2) ------> [ partition buffer: 1 2     ]      |
produce(3) ------> [ partition buffer: 1 2 3   ]      |
   |                       |                           |
   |  accumulation is bounded by these three values:  |
   |    time  : linger.ms          = 5 ms             |
   |    size  : batch.size         = 1,000,000 bytes  |
   |    count : batch.num.messages = 10,000           |
   |                       |                           |
   |       [ batch 1 ] === ProduceRequest ==========> |
   |                       | <========== ack ========= |
   |                       |                           |
poll(0)  serves callbacks that are already ready      |
flush()  serves callbacks until drained or timeout    |

Time

linger.ms=5 allows a short accumulation window; more waiting may improve throughput but adds latency.

Bytes

batch.size=1,000,000 limits the MessageSet size, including protocol overhead.

Count

batch.num.messages=10,000 is a librdkafka-specific record-count ceiling.

Do not import Java defaults. The Java producer’s commonly cited 16 KB batch.size is not this Python client’s default.

Restored · producer batching 3 of 3

Batching amortizes overhead; waiting can add latency

Fewer requests and better compression reduce overhead; waiting to accumulate records can delay an individual record.

Offset management

Three commit paths, three trade-offs

Automatic versus manual controls when progress becomes eligible; sync versus async controls whether the caller waits for the broker response.

Auto commit
  • enable.auto.commit=true
  • Stored offsets commit periodically in the background.
  • Simple, but processing may still be unfinished.
  • Demo 03A makes this behavior observable.
Manual sync
  • Disable auto commit and auto offset store.
  • commit(..., asynchronous=False)
  • The call waits and returns partition results.
  • Clear acknowledgement; more blocking.
Manual async
  • Same post-processing commit point.
  • commit(..., asynchronous=True)
  • The call returns before acknowledgement.
  • on_commit records success/failure.

All three store the next offset to consume. The difference is when progress is selected and whether your code waits for confirmation.

Consumer lifecycle

The poll loop is small—but the order matters

The course demos are intentionally bounded so every run finishes and writes inspectable evidence.

consumer.subscribe([TOPIC_NAME], on_assign=...)

try:
    while len(records) < max_messages:
        message = consumer.poll(poll_timeout)
        if message is None:
            continue
        if message.error():
            handle_or_raise(message.error())

        record = message_to_record(message)
        process(record)
        if manual_commit:
            commit_after_success(message)
finally:
    consumer.close()

poll()

Returns one message, no message before timeout, or a consumer error/event.

Finite limits

max_messages, idle timeout, and run timeout prevent classroom hangs.

close()

Stops consuming, leaves the group, releases resources, and commits stored offsets only when auto commit is enabled.

Consumer close is not producer flush. Consumers have no outgoing delivery queue to drain.

Exit ticket

If you can predict the next offset, you are ready

Answer these four questions before every consumer run.

Where will this group start?Use a valid commit first; otherwise apply the reset policy or explicit replay.
Who reads each partition?At most one active member per partition inside one group; other groups are independent.
When is progress safe?After decode, validation, and successful processing; commit the next offset.
Which Python API fits?Standard Consumer for ordinary scripts; native AIO when Kafka shares an event loop.
