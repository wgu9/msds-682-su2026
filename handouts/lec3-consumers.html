Retention

Consumers read; retention deletes

Reading a record does not remove it from Kafka. Multiple applications can read the same retained history.

Consumer

Fetches a copy

The client receives record bytes and metadata. The topic log remains intact.

Consumer group

Stores progress

Committed offsets record where one logical application should resume.

Broker policy

Removes old data

Time/size retention—or compaction policy—controls what Kafka eventually deletes.

Why AI engineers care: feature pipelines, monitoring, audit, and RAG-index updates can use separate groups to read the same event stream independently.

Two positions

Position and committed offset are different

One describes the current process; the other is the group’s durable recovery point.

0
1

Committed offset = 3

Recovery starts at offset 3

Offsets 0–2 are durably acknowledged as complete. Kafka stores the next offset to read.

Current position = 6

This process has fetched through 5

If it crashes now, offsets 3–5 may be delivered again because the durable position is still 3.

Python convenience: consumer.commit(message=msg) commits msg.offset() + 1.

Starting position

auto.offset.reset is only a fallback

Kafka checks for a valid committed position first.

Valid committed offset? For this group, topic, and partition
→
Yes → resume thereauto.offset.reset is ignored.
No → apply the reset policyearliest, latest, or error.
earliest

Start at the oldest retained record.

latest

Start at the current log end and wait for future records.

error

Raise an error so the application decides what to do.

Direct rule: earliest is not a rewind button. Use explicit replay when you intend to override normal group progress.

Delivery semantics

Process successfully, then commit

The commit point is an application decision about completed work—not merely fetched bytes.

poll
→
check error
→
decode UTF-8
→
validate TripEvent
→
process successfully
→
commit offset + 1

Commit too early

A crash can skip unfinished work

The next consumer starts after a record that never completed application processing.

Commit after success

A crash may repeat work

Reprocessing is visible and manageable; idempotent application logic handles duplicates.

Demo 03B contract: disable automatic offset storage, decode and validate first, then request the commit.
13 / 25

Offset management

Three commit paths, three trade-offs

Automatic versus manual controls when progress becomes eligible; sync versus async controls whether the caller waits for the broker response.

Auto commit
  • enable.auto.commit=true
  • Stored offsets commit periodically in the background.
  • Simple, but processing may still be unfinished.
  • Demo 03A makes this behavior observable.
Manual sync
  • Disable auto commit and auto offset store.
  • commit(..., asynchronous=False)
  • The call waits and returns partition results.
  • Clear acknowledgement; more blocking.
Manual async
  • Same post-processing commit point.
  • commit(..., asynchronous=True)
  • The call returns before acknowledgement.
  • on_commit records success/failure.

All three store the next offset to consume. The difference is when progress is selected and whether your code waits for confirmation.

Consumer lifecycle

The poll loop is small—but the order matters

The course demos are intentionally bounded so every run finishes and writes inspectable evidence.

consumer.subscribe([TOPIC_NAME], on_assign=...)

try:
    while len(records) < max_messages:
        message = consumer.poll(poll_timeout)
        if message is None:
            continue
        if message.error():
            handle_or_raise(message.error())

        record = message_to_record(message)
        process(record)
        if manual_commit:
            commit_after_success(message)
finally:
    consumer.close()

poll()

Returns one message, no message before timeout, or a consumer error/event.

Finite limits

max_messages, idle timeout, and run timeout prevent classroom hangs.

close()

Stops consuming, leaves the group, releases resources, and commits stored offsets only when auto commit is enabled.

Consumer close is not producer flush. Consumers have no outgoing delivery queue to drain.

Concurrency decision

Kafka asynchrony is not Python asyncio

They solve different waiting problems.

Standard Producer librdkafka

produce()
enqueue
→
background
I/O threads
→
broker +
delivery result

poll()/flush() serve Python delivery callbacks. No event loop is required.

Python application asyncio

Kafka AIO
task
↔
one event
loop
↔
HTTP / DB /
other async I/O

Use native AIO clients when Kafka must coexist with other nonblocking work in an existing event loop.

Never claim: putting blocking Consumer.poll() inside async def makes it nonblocking. Ordinary CLI consumers should use the standard API.

For CPython 3.11, threads are mainly useful for blocking I/O or libraries that release the GIL; pure-Python CPU parallelism usually needs processes or native/vectorized code.

Demo 03D · Native AIO

03D waits for assignment, not a guessed sleep

The producer starts only after Kafka confirms the consumer’s partition assignment.

from confluent_kafka.aio import \
    AIOConsumer, AIOProducer

assignment_ready = asyncio.Event()

async def on_assign(consumer, partitions):
    await consumer.assign(partitions)
    assignment_ready.set()

await asyncio.wait_for(
    assignment_ready.wait(), timeout=15
)

AIOConsumer subscribes.
The group coordinator assigns partitions.

on_assign sets the event.
Assignment evidence is recorded first.

AIOProducer begins.
It produces six records, then both tasks finish through asyncio.gather().

Both clients close.
Timeout and cancellation paths still release resources.

python demo03d_confluent_asyncio_produce_consume.py
--count 6 --assignment-timeout 15

Exit ticket

If you can predict the next offset, you are ready

Answer these four questions before every consumer run.

Where will this group start?Use a valid commit first; otherwise apply the reset policy or explicit replay.
Who reads each partition?At most one active member per partition inside one group; other groups are independent.
When is progress safe?After decode, validation, and successful processing; commit the next offset.
Which Python API fits?Standard Consumer for ordinary scripts; native AIO when Kafka shares an event loop.
